2) Matchmaker decision tree (exists vs new + next version)

This is the “factory brain.” It should be deterministic and idempotent.

Inputs

Upcoming event card (from ESPN/UFC schedule ingest)

Bouts list: (Real Fighter A name, Real Fighter B name, date, event ref)

Outputs

Canonical DB records (event, bouts)

Job emissions for:

new fighter creation if missing

version creation for existing fighters

metadata generation linking A vs B and event ref

Decision tree
FOR EACH EVENT (E):
  Normalize E into canonical event_ref, date_time, bout list

  Upsert EVENT in DB:
    event_id = hash(event_ref + date_time)

  FOR EACH BOUT (A_name vs B_name):

    A = RESOLVE_REAL_FIGHTER(A_name)
    B = RESOLVE_REAL_FIGHTER(B_name)

    IF A not found in Fighters table:
        CREATE fighter_id for A
        CREATE character mapping for A (character_id)
        EMIT JOB: CREATE_FIGHTER_IF_MISSING(fighter_id, A_name)
        (optionally EMIT JOB: CREATE_AGENT_FOLDER for A)

    IF B not found in Fighters table:
        same as above for B

    // At this point both fighters exist (or are pending creation)
    // Versioning logic is deterministic:
    A_next_version = (max(version) for A.fighter_id) + 1
    B_next_version = (max(version) for B.fighter_id) + 1

    // Create the “appearance/version” records (idempotent)
    Upsert FIGHTER_VERSION row for A:
      (fighter_id=A, version=A_next_version, event_id=E, opponent=B, bout_id)
    Upsert FIGHTER_VERSION row for B:
      (fighter_id=B, version=B_next_version, event_id=E, opponent=A, bout_id)

    EMIT JOBS (idempotent keys):
      - BUILD_METADATA(fighter_id=A, version=A_next_version, opponent=B, event_ref)
      - BUILD_METADATA(fighter_id=B, version=B_next_version, opponent=A, event_ref)

      - BUILD_ART_PROMPTS(...) for A and B (if needed)
      - PIN_METADATA(...) after metadata build
      - PREP_SALES_CONFIG(event_id, fighter_id, version, tiers=SE/LE/FE)
      - (optional) SCHEDULE_SOCIAL(event_id, fighter_id, version)
Key rule: “idempotent by unique_key”

Every job has a unique_key so reruns don’t duplicate versions or sales:

unique_key = event_id + fighter_id + version + job_type

If Matchmaker runs twice, it re-discovers the same event and produces no duplicates.

Matchmaker Spec V1
Purpose

Matchmaker ingests upcoming UFC events, normalizes them into a canonical DB model, resolves real fighters → Fight Farm character fighters, determines next version numbers, and emits idempotent jobs for downstream workers (NFTBuilder, SaleDeployer, Social, etc.).

Matchmaker does not mint. It only creates/updates records + emits jobs.

A) Canonical IDs and Idempotency
1) Canonical identifiers

Use deterministic IDs so repeated runs don’t create duplicates.

Event ID

event_id = sha256(chain="UFC" + "|" + event_ref + "|" + event_date_utc)

Bout ID

bout_id = sha256(event_id + "|" + normalized_real_name_A + "|" + normalized_real_name_B + "|" + weight_class)

Fighter ID (internal Fight Farm character fighter id)

fighter_id = "FF_" + base32(sha256(normalized_real_fighter_name))[:10]

immutable once created

Version job unique keys

unique_key = sha256(job_type + "|" + event_id + "|" + fighter_id + "|" + version)

2) Idempotency rules (hard requirements)

Every “create” operation is an upsert.

Every emitted job includes unique_key and must be inserted with a unique constraint.

If a job already exists, do nothing.

This ensures Matchmaker can run hourly without duplication.

D) Matchmaker algorithm (implementation details)
Step 1: Ingest and normalize event data

Pull schedule

Normalize:

event_ref, start_time_utc, bout list

normalize real fighter names into real_name_norm

Step 2: Upsert event + bouts

Upsert events

For each bout, upsert bouts

Store real_name_*_norm always

Step 3: Resolve fighters (exists vs new)

For each bout fighter:

SELECT fighter_id FROM fighters WHERE real_name_norm = ?

If not found:

create fighter_id deterministically

insert fighter row (or emit CREATE_FIGHTER_IF_MISSING)

emit CREATE_FIGHTER_IF_MISSING job

Then update bouts.fighter_id_a/b as soon as resolved.

Step 4: Determine next version number

For each resolved fighter:

SELECT max(version) FROM fighter_versions WHERE fighter_id = ?

next_version = (max_version or 0) + 1

Step 5: Upsert fighter_versions rows

Create one fighter_versions row per fighter appearance.

(fighter_id, version) must be unique.

include opponent_fighter_id, event_id, bout_id, event_ref, date

Step 6: Emit idempotent jobs

Emit the standard pipeline:

BUILD_METADATA (base + tiers)

GENERATE_ART_PROMPT

PIN_METADATA

PREP_SALES_CONFIG

DEPLOY_SALES (or separate approval gate)

All with unique_key constraints.

Naming & Normalization Spec V1
Goal

Convert any real-world fighter name string into a stable canonical key (real_name_norm) so:

“José Aldo” = “Jose Aldo”

“Robert ‘The Reaper’ Whittaker” = “Robert Whittaker”

“Jon Jones” ≠ “Jonathan Jones” (unless you explicitly alias it)

the Matchmaker never creates duplicate fighters due to formatting

This spec produces:

real_name_display (pretty, human)

real_name_norm (canonical, DB unique key)

real_name_fingerprint (optional hash key, if you want)

1) Storage rules (DB fields)

fighters table

real_name_norm = canonical normalized string (unique)

optional:

real_name_aliases[] (array of alternative norms)

bouts table

real_name_a_norm, real_name_b_norm

2) Normalization pipeline (strict order)

Given input string s:

Step 0 — Trim and unify whitespace

trim leading/trailing whitespace

replace any internal multiple spaces/tabs/newlines with a single space

Step 1 — Remove bracketed and quoted nicknames

Remove nicknames and clutter commonly inserted by sources:

content in parentheses ( ... )

content in quotes " ... " and ' ... ' when it’s clearly a nickname segment

content between fancy quotes “ ... ” and ‘ ... ’

Examples:

Robert "The Reaper" Whittaker → Robert Whittaker

Alexa Grasso (Champion) → Alexa Grasso

Keep apostrophes that are part of names (O’Malley), but strip quoted nickname segments.

Step 2 — Unicode normalize + strip diacritics

apply Unicode NFKD

remove combining marks

Examples:

José → Jose

Valentina Shevchenko stays same

Step 3 — Convert to lowercase

Max Holloway → max holloway

Step 4 — Standardize punctuation

Replace these with a simple ASCII equivalent:

curly apostrophe ’ → '

en/em dashes –/— → -

Then remove punctuation except:

hyphen -

apostrophe '

period . (only if used in initials like “J.J.”)

Examples:

Sean O’Malley → sean o'malley

Jean-Luc stays jean-luc

Step 5 — Remove suffixes / honorifics

Strip common suffix tokens at end:

jr, jr.

sr, sr.

ii, iii, iv

Strip titles at beginning if they appear (rare):

mr, ms, dr

Examples:

Kamaru Usman Jr. → kamaru usman

Bruno Silva II → bruno silva

Step 6 — Collapse whitespace again

Make sure it’s single spaces.

Output

real_name_norm = result

3) Allowed character set (enforced)

After normalization, real_name_norm must match:

lowercase letters a-z

digits 0-9 (rare but allowed)

space

hyphen -

apostrophe '

period . (initials only)

^[a-z0-9\s\-\.'’]+$

(We convert ’ to ' earlier, so ideally only ' remains.)

Here’s Versioning Spec V1 — the rulebook for how versions increment, what happens when fights change, and how to keep everything consistent without manual chaos. This is the one that prevents “version drift” and accidental double-mints.

Versioning Spec V1
Goal

Assign NFT version numbers deterministically so that:

every fighter appearance on a scheduled UFC event produces at most one new version

version numbers never collide

re-runs of Matchmaker don’t create duplicates

edge cases (canceled fights, replacement opponents, rebookings) don’t break the system

metadata can always link fighter A vs fighter B + event_ref

1) Core invariants (non-negotiable)
Invariant A — One version per fighter per event

A fighter can only have one version tied to a given event_id.

✅ allowed: fighter appears once on the card
❌ not allowed: a fighter gets two different versions for same event

Enforce with DB constraint:

UNIQUE (fighter_id, event_id)

Invariant B — Versions are monotonic per fighter

For each fighter:

version starts at 1

version increments by +1 for each new event appearance that becomes “committed”

Enforce with DB constraint:

UNIQUE (fighter_id, version)

Invariant C — Version assignment is idempotent

If Matchmaker is rerun, it must produce the same version record, not a new one.

Mechanism:

Matchmaker uses upsert on (fighter_id, event_id)

If record exists, reuse that version

2) Version assignment states (lifecycle)

Each fighter-version record moves through states:

planned

metadata_draft

pinned

sale_live

minted

archived (for cancellations / never went live)

These states matter because version assignment should become “locked” once you go live.

3) When do you “commit” a version number?
Commit point (recommended)

A version becomes committed when you approve DEPLOY_SALES (or when sales go live).

Reason:

schedules change

opponents change

bouts move cards
If you commit too early, you create version gaps or duplicates.

Rule

Before sale_live, versions can be reconciled.

After sale_live, version is locked and never reassigned.

4) The version assignment algorithm (deterministic)

Given a fighter F and event E:

Step 1 — Check if fighter already has a version for this event

Query:

SELECT version FROM fighter_versions WHERE fighter_id=F AND event_id=E

If exists:

use it. Done.

If not:

create a candidate record but do not finalize until commit point.

Step 2 — Determine next version number

Query:

SELECT max(version) FROM fighter_versions WHERE fighter_id=F

Candidate:

candidate_version = max_version + 1 (or 1 if null)

Insert:

(fighter_id=F, version=candidate_version, event_id=E, status='planned')

Because of UNIQUE constraints:

reruns can’t create duplicates.

5) Handling opponent changes (the big edge case)
Scenario A: opponent changes before sale_live (most common)

Example:

Fighter A vs Fighter B scheduled

Fighter B pulls out

Fighter C replaces B

You have not deployed sales yet

Rule

Keep the same version number for Fighter A for that event.
Update:

opponent_fighter_id in the existing fighter_versions row

Why: the version represents the fighter’s appearance on the event, not the opponent identity.

So:

A’s version stays Vn

you update metadata job payloads and regenerate images if needed

Scenario B: opponent changes after sale_live

This is trickier because NFTs might already exist.

Rule (V1 practical)

Do not create a new version

keep version as-is

update “opponent” fields in off-chain DB and on the frontend

optionally set an on-chain metadata update if you support dynamic metadata (later)

You can also add a metadata field:

opponent_status: "original"|"replacement"

6) Handling fight cancellation
Scenario A: canceled before sale_live

Rule:

mark fighter_versions.status = archived

do not deploy sales

do not mint

Version number will create a “gap” only if you had already created it—this is why commit point is important.

Best practice: if canceled early, delete the planned record OR archive it but do not advance max_version for future, by marking it archived and excluding archived from max-version query.

Important technical rule

Max version query should ignore archived records unless you explicitly want gaps.

Recommended:

max(version) WHERE status != 'archived'

Scenario B: canceled after sale_live

Rule:

you cannot un-mint history

treat the NFTs as “event canceled”

set bout_status = canceled in metadata

this becomes collectible history (and can be rare/valuable)

7) Handling rebooked fights (same matchup later)

Example:

A vs B booked for UFC 305

canceled

rebooked for UFC 307

Rule

UFC 305 version is tied to UFC 305 only

the rebooked fight is a new event, therefore a new version

A gets Vn at UFC 305 (archived or minted depending on timing)

A gets Vn+1 at UFC 307

8) Handling replacements on short notice (new fighter not in DB)

If replacement fighter isn’t in your DB:

Rule:

create fighter record (or emit CREATE_FIGHTER_IF_MISSING)

assign their version as normal for the event

rebuild bout metadata

This is why Matchmaker needs a clean “exists vs new” branch.

9) One fighter appearing twice in close windows (rare scheduling changes)

Sometimes fighters get rebooked quickly or appear on consecutive cards.

No special case needed:

each event creates one new version

monotonic sequence continues

10) Tier behavior (SE/LE/FE)

Tier does not affect version numbers.

A fighter/version produces:

SE Vn (mint on-demand during event window)

LE Vn (up to 5)

FE Vn (1-of-1)

All share:

same fighter_id, version, event_ref, bout linkage

11) Required constraints (must implement)

In DB:

UNIQUE (fighter_id, version)

UNIQUE (fighter_id, event_id) ✅ prevents double-versions per event

UNIQUE (event_id, bout_id) or bout uniqueness constraint

In jobs:

UNIQUE (unique_key) ✅ idempotent job creation

12) Recommended “approval gate” rule

To avoid bad scrapes producing bad versions:

Matchmaker emits DEPLOY_SALES with approved=false

A human/agent reviewer toggles to approved=true

Only then SaleDeployer deploys minters/launchpads/auctions

This is low effort and high safety.

The core problem

You want an NFT to remain:

usable for upgrades

How it works

NFT tokenURI points to a resolver contract (or a simple base URI + tokenId that resolves through a contract).

The resolver returns a metadata URI based on current state:

base immutable metadata

plus a small “context record” you can update

FightContextRegistry (simple mapping)

“Final opponent: Y (replacement)”

“Bout canceled” if needed

Pros

transparent

verifiable

no “trust me bro” off-chain edits

marketplaces that refresh metadata will display updates

Cons

you must build resolver/registry logic
